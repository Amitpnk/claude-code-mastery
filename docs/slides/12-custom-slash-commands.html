What is a custom command?

The built-in set from Video 4 — now with your own added in.
✓
A prompt template you invoke by name
Type /yourcommand and the full instruction runs — nothing retyped.
✓
Lives as a markdown file
Arguments are optional — a command can take input or run as-is.
✓
The natural next step after Skills
Skills package knowledge; commands package the exact prompt you keep retyping.

Anatomy of a good command

 
.claude/commands/review.md
# Command: review
# Short description shown in autocomplete
Review the current diff for:
- Logic errors and edge cases
- Consistency with our error-handling convention
- Anything that touches auth or payments — flag explicitly

# $ARGUMENTS optionally passed after the command

Global vs project commands

 

Global

~/.claude/commands
  • Your personal habits, every project you touch
  • Not shared — just for you

Project

.claude/commands, committed to git
  • Team-shared workflows specific to this repo
  • Everyone gets the same command, same behavior

That repetition is the signal — turn it into a command immediately.

Building one

 
Notice a repeated prompt
typed 3+ times
→
Write it as a command
.md file, clear name
→
Test it
run it for real
→
Refine the wording
if output drifts
→
Commit it
if project-scoped

Common mistakes

 

Do this

  • Build a command the moment you notice real repetition
  • Keep each command scoped to one clear job
  • Commit project commands so the whole team benefits

Not this

  • Building commands speculatively, before you've repeated anything
  • One giant command trying to do five unrelated things
  • Leaving a team-useful command sitting only in your global folder

From your prompts to your systems

 

Custom commands package the prompts you already know how to write. The next step is connecting Claude to systems it can't reach on its own — your database, your GitHub, your internal tools. That's MCP, next video.

Key takeaways

 
✓

A custom command packages a prompt you keep retyping into /yourcommand.

✓

A good command has a clear name, a specific job, and optional arguments.

✓

Global commands are personal habits; project commands are team-shared, committed to git.

✓

The signal to build one: you've typed the same prompt three times.
