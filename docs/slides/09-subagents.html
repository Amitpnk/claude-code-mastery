The scale problem

 

A single long session accumulates context, cost, and drift the longer it runs — exactly what Video 7 warned about. One giant agent trying to research, implement, and verify everything in one context window gets expensive and loses focus.

What is a SubAgent?

A separate, scoped agent instance — not another chat tab.
✓
Its own context window
Whatever it reads or explores doesn't touch your main session's context.
✓
Spawned for a specific task
You send it off scoped to one job, not a whole open-ended project.
✓
Reports back a result
It returns a summary — not the entire transcript of everything it did.

When to use SubAgents

 
01
Research tasks
Exploring a large codebase without bloating your main session's context.
02
Parallel independent work
Several unrelated searches or investigations running at once.
03
Isolating risky or exploratory work
Keep experimentation out of your main thread's context and history.
04
Protecting your main context window
Large outputs land in the SubAgent's context, not yours.

Single session vs parallel SubAgents

 

Single long session

Everything in one context
  • Simpler and cheaper for small, focused tasks
  • Gets expensive and slow to steer at scale

Parallel SubAgents

Isolated contexts per task
  • Faster for independent, parallelizable work
  • Costs less overall on large, multi-part jobs

Common mistakes

 

Do this

  • Scope each SubAgent to one narrow, well-defined task
  • Use them for genuinely independent, parallelizable work
  • Let the orchestrator stay focused on synthesis, not detail work

Not this

  • Spawning a SubAgent for a trivial one-line task
  • Duplicating the same work across the main session and a SubAgent
  • Giving a SubAgent a vague, open-ended goal instead of a scoped one

From delegation to design

 

Delegating work well is only half the discipline — knowing what to delegate, and why, comes from having a clear spec in the first place. That's exactly where we go next.

Key takeaways

 
✓

A SubAgent runs in its own context window and reports back a summary, not a transcript.

✓

The orchestrator pattern keeps your main session clean while work happens elsewhere.

✓

Use SubAgents for research, parallel work, and isolating risk — not trivial one-liners.

✓

Parallel SubAgents beat a single long session specifically when tasks are independent.
