The core discipline

Three habits, applied every single session.
✓
Small, scoped changes
One logical change per request — not a whole feature in one shot.
✓
Review every diff
Read what changed before you move to the next step, not after three more.
✓
Commit before you start
A clean git state is the checkpoint you fall back to if something goes wrong.

Before & after

 
"Build the whole feature"
→
One giant diff — unreviewable, and you're guessing what changed and why.
One logical change
→
A diff small enough to actually read and verify.
Scoped task + review
→
Code you understand, and can maintain, at every step.

Reviewing diffs like a PR

Treat every Claude Code change like a colleague's pull request.
✓
Read every changed file
Not just the summary Claude gives you — the actual diff.
✓
Question anything surprising
A file you didn't expect to change is a signal to stop and ask why.
✓
Run tests before moving on
Confirm the change works before stacking the next request on top of it.

Version control as your safety net

 
01
Commit before every session
A clean starting point you can always fall back to.
02
Small commits
Each one is a checkpoint you can revert without losing everything else.
03
Branch per feature
Isolates risk from main — a bad session can't take down your working branch.
04
Never work uncommitted
An uncommitted mess across multiple files is very hard to unwind cleanly.

Permission modes, revisited

From Video 2 — the mode you're in decides how much review actually happens.

Default mode

Prompts before each new tool use
  • Forces a review checkpoint at every step
  • Slower, but nothing lands without your eyes on it

acceptEdits mode

Auto-accepts file edits
  • Faster once you trust the workflow
  • Only appropriate after you've built that trust — not from day one

Common failure modes

 

Do this

  • Stop and ask when Claude seems uncertain
  • Review before you accept, every time
  • Test after each scoped change

Not this

  • Approving edits without reading them
  • Letting unrelated changes stack up across requests
  • Skipping tests "just this once"

Key takeaways

 
✓

One logical change per request beats one giant unreviewable diff.

✓

✓

✓

The golden loop: commit → scoped prompt → review → test → commit.
