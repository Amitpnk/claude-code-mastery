A spec is where you agree on "done"

 

Back in Video 1, we said Claude Code is a brilliant junior developer that needs clear instructions. A spec is exactly that: the moment you and Claude agree on what "done" actually means — before either of you starts writing code.

What goes into a good spec

 
01
The problem
What's broken or missing, stated in plain terms.
02
The scope
What's in — and just as important, what's explicitly out.
03
Acceptance criteria
How you'll actually know it's done, not just "it works."
04
Constraints
Technical, UX, or business limits the solution has to respect.

Where a spec lives

Not in chat history. In your repo.
✓
A markdown file in the repo
Not buried in a conversation that scrolls away and disappears.
✓
Referenced by path, not retyped
Point Claude at the file instead of re-explaining it every session.
✓
Versioned with the code
It evolves alongside the feature it describes, tracked in git.

Common mistakes

 

Do this

  • Write acceptance criteria you can actually test against
  • State what's explicitly out of scope
  • Keep the spec in the repo, referenced by path

Not this

  • Treating a one-line prompt as a spec
  • Leaving scope implicit and hoping Claude guesses right
  • Letting the spec live only in chat history

Spec defines what. Plan defines how.

 

A spec settles what "done" means. It doesn't settle how you'll get there — that's a separate decision, and it's exactly what Plan Mode and Ultraplan Mode are for, coming up next.

Key takeaways

 
✓

A spec is where you and Claude agree on "done" before any code exists.

✓

Good specs define the problem, the scope, acceptance criteria, and constraints.

✓

Keep it in the repo as a file, referenced by path — not buried in chat history.

✓

A clear spec is what eliminates the drift we've been fighting since Video 1.
